Quaid is a JavaScript library that works with jQuery to help you write cleaner, more-robust code. You can think of it as a little framework with which to easily build widgets and other stateful jQuery plugins. Quaid provides two main services: a more modular way to write complex JavaScript applications with jQuery, and a robust way to handle errors from async server requests. Also included for convenience is multi-level, cross-browser logging with failover, a small framework for data formatting, and form handling utilities.

Quaid grew pretty organically over a period of about a year and a half at AdRoll. So while it may seem like a kind of random smattering of tools, they work well together to make a solid minimum tool set for your app.

Dependencies

jQuery is the main dependency. If you use Quaid's form module, you will need Mike Alsup's form plugin, and if you want Quaid's validation module to integrate into your client side validaiton scheme, you need the Bassistance validation plugin.

Quaid's six modules are not all necessary. Only core, log and util are totally necessary. You can build your own distribution with the build.py script in the source tree's root.

# will build the minimum set
python build.py core log util

# will build all
python build.py

Big Fat Example

I'll start with a pretty comprehensive example so you can get a feel for it.

Assume you have to build an async form. You want your real validation to be done on the server, and maybe some superficial convenience stuff to be checked on the client side. The user shouldn't see the client and server validation as any different, and he needs feedback when the page is waiting for the server. When the form validates, you want the data returned from the server to update some markup. You have a lot of forms with these requirements. What do you do?

Notice that the required validation check is done by the client, but all other validation is done on the server. The neat thing is that both validation issues from the server and the client run through the same mechanism in the Bassistance validation plugin.

// We create a plugin called UpdateForm that extends
// Quaid provided Q.AsyncForm
Q.AsyncForm.extend('UpdateForm', {
    
    // You can specify static members this way.
    // For example, these defaults will be available via
    // Q.UpdateForm.defaults
    defaults: {
        blockToUpdate: null,
        dataFormats: {},
        loaderLocation: {position: 'absolute', right: 5, top: 5}
    }
    
}, {
    
    // Our plugin's constructor
    init: function(container, settings){
        
        // We call Q.AsyncForm's constructor passing it settings
        // which may be overriden by the user.
        settings = $.extend({}, this._class.defaults, settings)
        this._super(container, settings)
        this._block = $(this.settings.blockToUpdate);
    },
    
    // Override Q.AsyncForm's onSuccess function
    onSuccess: function(data){
    
        // We loop through the results, format the data according to
        // the formats specified in the settings, then we put the
        // server's data into the markup.
        for(var k in data.results){
            
            // Format the raw data from the server based on settigns
            var formattedData = Q.DataFormatters.get(
                                    this.settings.dataFormats[k],
                                    data.results[k]);
            
            // Server's key matches some element's class name.
            this._block.find('.'+k).text(formattedData);
        }
        
        //call the user's callback
        this._super.apply(this, arguments); 
    }
});

// Here we instantiate our new plugin! 
$('#example1').UpdateForm({
    
    // Where to put the data from the server
    blockToUpdate: '#example1 .update-me',
    
    // Tells our plugin how to interpret the data
    // coming back from the server
    dataFormats: {
        mileage: 'decimal(1)',
        programmers: 'number'
    },
    
    // Client validation options. 
    validationOptions:{
        rules:{
            name: 'required',
            miles: 'required',
            programmers: 'required'
        }
    },
    
    // Our success callback
    onSuccess: function(data){
        this.container.find('.update-me').show();
    }
});

Philosophy

Writing easily maintainable code for the front end is difficult. How do you properly abstract a chunk of HTML? How do you write it in such a way that the code makes sense in a month when you come back to it? How about writing it so another human can make sense of it and potentially extend it?

I feel the problem in writing good code for the front end is a problem of encouragement. I often feel I am encouraged to be sloppy. jQuery, while amazing, encourages me to write simple plugins that store state in the DOM via $.data(), then chain them together and bury them in layers of callbacks. This is problematic for maintinence and debugging.

My goal with Quaid is to encourage you to build plugins a different way. Instead of applying a chain of plugin calls to some element or collection of elements, Quaid helps you create a meaningful, consistent API for your chunk of markup.

The more detailed core docs walk through building a basic plugin that toggles markup. With Quaid, the resulting plugin is simple, readable, flexible, and extensible.

Module Overview

Quaid is broken down into six modules.

Core

The core implements the Class pattern and a mechanism for connecting your shiny new classes to jQuery. Here's how you might create a toggler with Quaid.

Q.Toggler = Class.extend('Toggler', {
    init: function(container, settings){
        var defaults = {
            trigger: null, // give me a string or jQuery object
        };
        this._super(container, $.extend({}, defaults, settings));
        
        $(this.settings.trigger).click(function(){
            self.toggle();
            return false;
        });
    },
    
    close: function(){this.container.hide();},
    open: function(){this.container.show();},
    
    toggle: function(){
        if(this.container.css('display') == 'none')
            this.open();
        else
            this.close();
    }
});
var mytoggler = $('#toggled1').Toggler({trigger: '#trigger1'});

// The result object has the functions you defined.
mytoggler.toggle();
mytoggler.open();
mytoggler.close();

Log

The log module defines a new log interface. It utilizes all the console[x]() functions and has failovers when they aren't available. For example, it can failover to firebug lite, or an even lighter 'SimpleConsole' that is included with Quaid. It defines new log methods

$.debug(); //level 0
$.info();  //level 1
$.warn();  //level 2
$.error(); //level 3
$.log();   //level 4

$.log.enabled = true;

The form module provides classes to encapsulate forms. Form handling shouldn't be a pain. This module eases the burn.

Util

What library would be complete without a util module? This one provides data formatting capabilities and functions I felt were missing from jQuery.

// parses a date
var date = $.parseDate('3.10.2010', 'm.d.Y H:M');

// formats data
// formatted = '1,011.482'

// formatted = '$1.48'
var formatted = Q.DataFormatters.get('dollar', 1.482389892);

Validation

The validation module handles errors from your server and connects your server's async responses to the Bassistance validation plugin..
